1. **_Process Identity_**: For the Go implementation, we followed a similar pattern to Elixir's mailboxes. For the most part, each process has one single channel (which we call a mailbox) that it uses to receive messages. These channels are created by the `ShopOwner` process for the Shop Owner, Barber, and the Waiting Room (lines `125-127`). Each Customer creates its own channel. The Barber is the only process to use multiple channels. The first channel, created and passed in by the Shop Owner, serves as its primary mailbox for all incoming messages. The second channel `RatingReplyChannel` (created on line `306`), is created by the Barber itself, as a dedicated channel to receive a rating from the customer who has just had their hair cut. Since only that customer is passed in this channel, it creates an isolated, one-shot reply without interference from other messages. When a message requires a response, such as a request for stats, the sender includes its own channel in the message struct.

2. **_State Management_**: It felt a bit different since Elixir uses a recursive loop, with each receive branch returning a new loop call with the updated state, making the transitions between states very explicit in the function arguments. State management through local variables felt more natural to us in Go, since we are more familiar with data changes inside a function rather than threading state through function arguments. Most of our programming experience has been in languages that also use this kind of style, such as Python. This being said, the waiting room logic felt a little more structured in Elixir, where pattern matching on the messages made it all flow easy, despite the extra ceremony of it all. In Go, the barber’s state felt very natural, since it was just changing the local variables within the event loop.

3. **_Sleeping Barber Handshake_**: In Elixir, the Barber’s behavior is handled by a separate `sleep_loop` function that only accepts `:wakeup` and `:shutdown` messages. This means that the Barber truly ignores all other messages while it is asleep. The waiting room then tracks whether the Barber is sleeping through a boolean thread through its recursive loop. It will send the `:wakeup` message to the Barber’s PID when a new customer arrives. In Go, the Barber's behavior is achieved through two different functions. The main loop `barber` (`283-346`) handles all waking behavior that the barber has. When the Barber receives a command to sleep (`MsgNoneWaiting`), it calls the second function `barberSleepLoop` and waits for it to return. `barberSleepLoop` sits and waits for a wake up message, or a shutdown message. When it receives a message, it passes a value back to the main loop, so that the main loop knows whether to exit as a final shutdown, or to resume waking behavior. It also receives the current running averages, and handles requests for statistics, to prevent a locking scenario where the Shop Owner is stuck waiting forever for statistics from a sleeping barber. Both of the languages approaches implement the same handshake.

4. **_Message Types_**: In Elixir, messages are just tuples or atoms with no boilerplate, like stated above. In Go, we defined a tag named `MsgKind` and a `Message` struct. Our receivers used `switch msg.Kind` to dispatch. This meant that we had to have more of a boilerplate, but it also collected the message shape into just one place. This made refactoring a bit easier since our compiler can catch the fields that were renamed. However, Elixir is safer because if it has a wrong tuple shape, it won’t match any clause to it handles bad messages, while Go doesn’t enforce that limitation and just “forgets” to fill in the relevant fields.

5. **_Select vs. Receive_**: We did not end up using `select` within our Go implementation. Every process reads from a single channel and not two separate ones, which is similar to Elixir’s single-mailbox model. If we had given the Barber two separate channels for the customer messages and the shutdown signals, then `select` would have been a better fit, but we felt like making a multiplexed channel with tagged messages was unnecessary for our implementation. So, instead we just switched to a separate loop that only accepts a certain kind of message on the same channel.

6. **_AI Tool Usage_**: For the Elixir implementation, we used AI tools to generate an initial working structure of the system using the actor model, asking it to use `spawn`, `send`, `receive` and to include the four required processes (Customer, Waiting Room, Barber, and Shop owner) with the correct message flow. The generated code helped set up the overall structure, separating each entity into its own process, passing states through recursive function calls, and avoiding shared variables. I had to make a few adjustments to get the final file functioning and compliant with assignment parameters. For instance, some helper functions were initially placed outside of their modules which led to syntax errors. To fix this, we had to restructure the code and adjust module boundaries. We also had to refine the sleeping and wakeup logic to make sure the waiting room correctly tracks whether the barber is asleep or not and only sends the wakeup signal when needed. Overall not many adjustments were required to ensure complete correctness.
